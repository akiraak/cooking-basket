# ログインコード再送への差し替え

## 目的・背景

`mobile/src/components/auth/AuthModal.tsx` の **コード入力ステップ**（`step === 'code'`）に
表示している「別のメールアドレスで試す」リンクを削除し、代わりに「ログインコードの再送」を
配置する。

現状の問題:

- 「別のメールアドレスで試す」はメール入力画面に戻すリンク。すぐ下に「キャンセル」も
  あり、押せばモーダルが閉じてやり直せるため、機能が重複している。
- メールが届かない／コードを失効させてしまった場合に、ユーザーが手元のままコードを再送する
  手段がない。「キャンセル」→ モーダル再表示 → メール入力 → 送信、という遠回りが必要。

「再送」を直接置けば、誤入力時のリカバリ動線が短くなり、UI も一段スッキリする。

## 対応方針

### 1. UI 差し替え（`mobile/src/components/auth/AuthModal.tsx`）

- `step === 'code'` ブロックから「別のメールアドレスで試す」(`handleBackToEmail`) を削除
- 同じ位置に「ログインコードの再送」リンクを追加
- 不要になった `handleBackToEmail` を削除
- 「キャンセル」リンクはそのまま残す（メール変更したい場合のリカバリ動線として機能する）

### 2. 再送ロジック

- 既存の `sendMagicCode(email)`（`/api/auth/login`）をそのまま再利用する
  - サーバ側 `createMagicLinkToken` は呼ばれるたびに既存の未使用トークンを `used=1`
    にしてから新しいコードを発行するので、再送 = 新コードに置き換わる
- 連打防止のためフロントでクールダウンを設ける（**30 秒**）
  - 状態: `const [resendCooldown, setResendCooldown] = useState(0)`
  - 再送成功時に 30 にセット、`setInterval` で 1 秒ずつデクリメント、0 で停止
  - `useEffect` の cleanup で interval を解放
  - クールダウン中はリンクテキストを「再送（残り N 秒）」に変えて `disabled`
- 完了フィードバックは軽量に Alert（`Alert.alert('送信しました', '新しいコードをメールで送信しました')`）
  - 失敗時は既存 `handleSendCode` と同じ Alert パターンを踏襲
- `loading` 中は再送ボタンも `disabled`（ベリファイ進行中の連打を防ぐため）
- モーダルを再表示する `useEffect`（visible 変化時の初期化）でも `resendCooldown` を 0 にリセット

### 3. ネーミング

- ハンドラ: `handleResendCode`
- リンクラベル（通常）: `ログインコードの再送`
- リンクラベル（クールダウン中）: `再送（残り {N} 秒）`

## 影響範囲

- 変更ファイル: `mobile/src/components/auth/AuthModal.tsx` のみ
- サーバ側変更なし（既存 `/api/auth/login` をそのまま使う）
- `mobile/src/stores/auth-store.ts` 変更なし（`sendMagicCode` を再利用）
- `mobile/src/api/auth.ts` 変更なし

## テスト方針

- 既存テスト: `mobile/__tests__/` には AuthModal の RN レンダリングテストは無い（CLAUDE.md
  記載どおり、モバイルのコンポーネント描画テストは未導入）。テストの追加は不要とする。
- 実機/Expo Go での手動確認:
  1. メール → コードを送信して `step === 'code'` まで進む
  2. 「ログインコードの再送」を押す → メール受信箱に新しいコードが届く
  3. 押下直後はリンクが「再送（残り 30 秒）」になり、disabled
  4. 30 秒後にラベルが元に戻り、再度押せる
  5. 1 通目のコードは無効化されている（`used=1`）ので、古いコードでログインしようとすると
     「コードが無効または期限切れです」になる
  6. 「キャンセル」を押すとモーダルが閉じ、再度ログインを開くと email ステップから始まる
- サーバ側は変更しないので Vitest の追加は不要
